Duplicated Redirect Function Code

functions/redirect.js ↔ cloudformation/template.yaml (lines 79–204)
Significant
Knowledge Leakage

The entire redirect function — validatePath, parsePath, buildQueryString, buildRedirectUrl, redirect301, notFound, and handler — exists in two independent copies:

  • functions/redirect.js — used by tests
  • cloudformation/template.yaml FunctionCode — the actually deployed version

The only difference is the KVS ID initialization ('KVS_ARN_PLACEHOLDER' vs '${RedirectKVS.Id}'). This is functional coupling in its most dangerous form: duplicated implementation with no explicit integration point.

Complexity Impact

Tests exercise functions/redirect.js, not the CloudFormation inline code. Tests can pass while the deployed function has a bug (or lacks a fix). The cause-and-effect relationship is hidden: "I changed the code and the tests pass, but production still behaves incorrectly."

Cascading Changes

Every change to redirect logic requires editing two files:

  • Adding a new redirect type (e.g., temporary 302 redirects)
  • Modifying path validation rules
  • Changing cache headers
  • Fixing a bug in wildcard URL construction
Recommendation

Extract the redirect function to a single source file and inject it into the CloudFormation template during deployment. The deploy.sh script can read functions/redirect.js, substitute the KVS placeholder, and inject it into the template before deployment. Tests continue to run against the source file.

Trade-off: Adds a build step. Worth it because it eliminates a silent failure mode where tests and production diverge.

UI AddLinkDialog.tsx ↔ CreateLink Lambda ↔ CloudFront Function
Minor
Knowledge Leakage

Three components independently implement knowledge of the wildcard redirect convention (/* suffix) and link validation rules:

The wildcard convention is implicit shared knowledge — not defined in a contract or schema.

Cascading Changes
  • Changing the wildcard convention requires updating three components across two runtimes
  • Adding a new redirect type (e.g., regex-based) would touch all three
  • Changing validation rules requires syncing the regex in both UI and Lambda
Recommendation

Acceptable technical debt given low volatility. The backend Lambda validation is authoritative; the UI validation is a UX convenience. If redirect types expand, consider storing the type explicitly in DynamoDB ({ key, value, type: "exact"|"wildcard" }) rather than encoding it in the value string.

Trade-off: Only worth doing if the redirect model actually needs to evolve.

CRUD Lambda Code Inline in CloudFormation

ListLinks, CreateLink, DeleteLink Lambdas embedded in template.yaml
Minor
Knowledge Leakage

Three Lambda functions containing application logic (validation, DynamoDB queries, error handling) are defined as inline ZipFile code within the 958-line CloudFormation template. This co-locates application concerns with infrastructure concerns.

Complexity Impact

The template serves dual purposes: infrastructure definition and application code host. Unlike the redirect function (which has a separate testable copy), the CRUD Lambda code exists only in the CloudFormation template.

Recommendation

For this size and volatility, inline Lambdas are pragmatic. If the CRUD logic grows more complex, extract the Lambda code to separate files (like the KVS sync Lambda already is) and reference them via S3Bucket/S3Key.

Trade-off: Extraction adds deployment complexity. The current approach is simpler. Only extract if Lambda code grows beyond simple CRUD.

Summary

This is a well-structured project for its scope. The system's overall low volatility (personal URL shortener, generic subdomain) means the identified coupling imbalances are unlikely to cause active pain.

The most actionable issue is the duplicated redirect function code, which creates a silent failure mode where tests and production can diverge — worth fixing regardless of volatility because it undermines test reliability.

The validation rule duplication and inline Lambda code are standard pragmatic trade-offs for a single-person, low-volatility project. They would become more significant if the system's scope expanded (e.g., multi-tenant, additional redirect types, team growth).
